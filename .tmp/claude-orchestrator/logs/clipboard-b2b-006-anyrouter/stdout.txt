## Clipboard file-detail / path behaviors that are now real

- Each file row in the detail pane is an independent clickable card. Clicking an existing file/directory calls `platform::shell::reveal_in_finder` (`open -R`), highlighting it in Finder.
- Per-row visual badge (`文件` / `目录` / `已不存在`) is computed from a single `file_path_states(...)` scan and reflects actual filesystem state at render time.
- Rows whose path no longer exists are muted, badged "已不存在", and have no click handler — they don't pretend the action would work.
- Below the preview the detail pane now shows **two** file-specific buttons in an action row:
  - "在访达中显示" wired to `reveal_first_existing_in_finder` (enabled iff any path exists)
  - "打开目录" wired to `open_selected_parent_dir` (enabled iff any path or parent dir exists)
- Each button honestly renders as disabled (muted color, no hover, no click) when there is no actionable target — replacing the previous single "打开目录" strip.
- Generic copy / pin / delete strip is preserved unchanged for all kinds.

## Clipboard behaviors still missing

- No per-row "open parent dir" — only whole-record reveal/open. Acceptable for a conservative batch; would need extra UI plumbing.
- No bulk reveal/open (e.g. reveal all paths). `open -R` accepts multiple args but the current button only handles first-match.
- No drag-source on file rows (suishou QML reference doesn't have this either).
- File rows don't differentiate icons by extension — all rows just say `文件` or `目录`.
- `history_store.rs` is still not renamed to `store.rs` as the migration guide eventually wants — kept out of scope here.
- File capture from text fallback (when osascript file-URL detection fails) is unchanged.
