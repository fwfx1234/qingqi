## image-compress batch behaviors now real

- **Background execution**: Batch compression runs on a spawned task, not in the UI render/click path. The UI stays responsive while files are being compressed.
- **Honest per-file status**: Items marked `Running` while being processed, `Success` only after the output file is actually written, `Failed` with real error messages on failure.
- **Coarse-grained progress**: No fake percentages. Status bar shows "正在压缩…" during work, and the result summary shows running/pending/success counts.
- **Result sync on render**: `collect_results()` drains completed results from shared state before each render, so the UI updates as results arrive.
- **Running guards**: `clear_items()` and `remove_item()` refuse during running state. Remove button hidden for running items.

## image-compress behaviors still missing

- **Per-entry live progress**: Individual file progress (e.g., "compressing file 3/7") is not shown. Only coarse "running" state.
- **Cancel/stop**: No way to abort a running batch. The worker runs to completion.
- **Retry in background**: `retry_entry` remains synchronous (single-file, fast enough for v1).
- **JobProvider integration**: Not implemented. The batch does not report to the shared job system.
- **Auto-refresh during batch**: The UI only refreshes when `render` is called (on user interaction or window focus). No periodic timer to poll shared state while the user is idle.
- **SaveAs dialog in background**: `rfd::FileDialog::save_file()` still runs on the main thread (as required by macOS).
