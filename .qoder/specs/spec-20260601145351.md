# Qingqi 项目全面优化计划（修正版）

## Context

经过代码审查，项目存在以下主要问题需要优化：

1. **FRECENCY_WEIGHT 常量重复定义**: 在 `plugin.rs`（`pub const`）和 `launcher.rs`（私有 `const`，注释注明需与 PluginManager 保持同步）各定义一次，值均为 5.0
2. **排序逻辑重复且有 Bug**: `PluginManager::sort_scored_commands` 和 `Launcher::query_commands` 实现了相同的三层排序算法，但 launcher 版本**遗漏了 `boost_map`（clipboard boost）项**，导致来自插件管理器的排序结果被 discard 后重新排序时丢失了 clipboard 提升权重
3. **Re-export 文件冗余**: `qingqi-app/src/core/` 下有 6 个文件使用显式 `pub use qingqi_plugin::xxx::{Type1, Type2, ...}` 进行类型转发，增加了维护成本
4. **缺少 workspace lint**: 没有统一的 clippy 规则控制代码质量
5. **icon_raster.rs 两份**: `crates/qingqi/icon_raster.rs` 和 `crates/qingqi-platform/src/icon_raster.rs` 几乎相同（前者多一行注释）
6. **大文件**: `launcher.rs` 1858 行，`window_controller.rs` 756 行；12+ 个 feature crate 文件超过 1000 行（其中 3 个超过 3000 行，均在 feature crate）
7. **unsafe 代码**: `clipboard.rs` 有 32 个 `#[allow(unsafe_op_in_unsafe_fn)]` 标注，30 个 `unsafe fn` 函数体内无 `unsafe {}` 包装块
8. **残余 `.unwrap()`**: 核心层生产代码仅 5 处 `.unwrap()`（4 处安全，1 处需修复）和 2 处 `.expect()`（均可接受）

> **注意**: 原版规范中声称的 "44 处 mutex 样板代码" 与事实不符 — `lock_or_recover` 已在核心层被广泛使用 52 次（window_controller.rs 31 次、launcher.rs 11 次、runtime.rs 7 次、background.rs 3 次），批量替换工作已基本完成。仅剩 `low_level_hook.rs:85` 一处真正的 mutex lock unwrap。

## 实施策略

本次（**阶段 1**）聚焦核心层（qingqi-core、qingqi-app、qingqi-plugin、qingqi-platform、qingqi-ui），暂不修改 feature crate。阶段 2 再处理 feature crate 的 view 拆分、unwrap 和 unsafe 清理。

- `qingqi-plugin`

### Step 2: Re-export 清理

**修改文件**（共 6 个）:
- `crates/qingqi-app/src/core/command.rs` — 改为显式导出
- `crates/qingqi-app/src/core/database.rs` — 改为显式导出

**现状**: 这些文件当前使用 `pub use qingqi_plugin::module::{Type1, Type2, ...};` 显式列出类型。问题在于它们仅作为转发层，增加了维护负担 — 每次上游 API 变化都需要同步更新。

**方案**: 评估是否可以直接让消费者 `use qingqi_plugin::...` 直接引用，从而**删除**这些转发文件。如果消费者大量依赖 `crate::core::xxx` 路径，则逐步迁移 import 后删除。

**改动模式**:

```rust
// 消费者文件中的 import 迁移示例
// 之前
use crate::core::command::{Command, CommandInvocation};
// 之后
use qingqi_plugin::command::{Command, CommandInvocation};
```

迁移完成后删除对应的 `crates/qingqi-app/src/core/{command,database,events,page,plugin_spec,storage}.rs` 文件，并从 `core/mod.rs` 中移除对应的 `pub mod` 声明。

### Step 3: 常量重复消除与排序逻辑统一

#### 3a. FRECENCY_WEIGHT 统一

**文件**:
- `crates/qingqi-core/src/plugin.rs:270` — 将 `const FRECENCY_WEIGHT: f64 = 5.0;` 改为 `pub`
- `crates/qingqi-app/src/app/launcher.rs:58-59` — 删除私有 `const FRECENCY_WEIGHT`，改为 `use qingqi_core::plugin::PluginManager;` 后通过 `PluginManager::FRECENCY_WEIGHT` 引用

#### 3b. 排序逻辑提取（含 Bug 修复）

**现状分析**:

`PluginManager::sort_scored_commands` 排序公式：
```
total = score + frecency * FRECENCY_WEIGHT + boost_from_clipboard_map
```

`Launcher::query_commands` 排序公式：
```
total = score + frecency * FRECENCY_WEIGHT   // 缺少 boost 项！
```

Launcher 先调用 `plugin_manager.query_commands_with_clipboard(query, limit, &boost_map)` 获取排序后的插件命令，然后与 app_catalog 结果合并后**丢弃插件排序、重新排序**。重新排序时使用的是与 PluginManager 几乎相同的三层比较器，但遗漏了 `boost_map` 项。这意味着 clipboard boost 影响了插件查询阶段的截断（哪些命令进入候选集），但不影响最终展示顺序。

**修复方案**:

在 `PluginManager` 中将排序逻辑提取为**公开关联函数**（不依赖 `&self`）：

```rust
// crates/qingqi-core/src/plugin.rs

impl PluginManager {
    /// 公开的排序函数，供 Launcher 等外部调用
    pub fn sort_commands(
        commands: &mut [(i32, Command)],
        usage_map: &HashMap<String, CommandUsage>,
        boost_map: &HashMap<String, i32>,
    ) {
        // 现有的 sort_scored_commands 逻辑，使用 Self::FRECENCY_WEIGHT
    }
}
```

在 `Launcher::query_commands` 中调用 `PluginManager::sort_commands(...)` 替代手写排序，同时传入 `&self.clipboard_boost_map` 修复 boost 遗漏的 bug。

### Step 7: 生产代码 .unwrap() 修复

**范围**: 核心层的生产代码（不含 feature crate，不含测试代码）

**实际统计**: 核心四 crate（qingqi-app、qingqi-platform、qingqi-core、qingqi-plugin）生产代码中仅 **5 处 `.unwrap()`** 和 **2 处 `.expect()`**（不含测试代码）。

| 文件 | 行号 | 表达式 | 分类 | 操作 |
|------|------|--------|------|------|
| `qingqi-platform/src/low_level_hook.rs` | 85 | `error_flag.lock().unwrap()` | Mutex lock | **需修复** |
| `qingqi-platform/src/power.rs` | 109, 110, 160, 161 | `CString::new("...").unwrap()` | CString 字面量 | 保持（不可能失败） |
| `qingqi-platform/src/hotkey.rs` | 42 | `manager.as_ref().expect("...")` | Option unwrap | 保持（合理 panic-on-bug） |
| `qingqi-app/src/app/app_index_store.rs` | 99 | `database.pool(key).expect("...")` | Result unwrap | 保持（合理 panic-on-bug） |

**唯一需修改的文件**: `crates/qingqi-platform/src/low_level_hook.rs:85`

// 之前

// 之后
*lock_or_recover(&error_flag, "error_flag") = Some(e);
// 或

### Step 8: clipboard.rs unsafe 清理

**文件**: `crates/qingqi-platform/src/clipboard.rs` (1348 行)

**现状统计**:

| 项目 | 数量 |
|------|------|
| `#[allow(unsafe_op_in_unsafe_fn)]` 标注 | 32 处 |
| 其中在非 `unsafe fn` 上（无效标注，直接删除） | 2 处 (line 825, 859) |
| 其中在 `unsafe fn` 上（需要处理） | 30 处 |
| `unsafe fn` 函数 | 30 个 |
| 现有 `unsafe {}` 块 | 26 个（均在普通 `fn` 中） |
| `unsafe fn` 函数内的 `unsafe {}` 块 | 0 个 |

**问题**: 所有 30 个 `unsafe fn` 函数体中直接使用 unsafe 操作（`msg_send!`、`std::mem::transmute`、`std::slice::from_raw_parts` 等），没有任何 `unsafe {}` 包装块，依赖 `unsafe fn` 的函数级上下文和 `#[allow]` 消除警告。

**预估工作量**: 删除 32 个 `#[allow]` 行，在 30 个函数中添加约 **70-90 个 `unsafe {}` 包装**。

**改动模式**:

```rust
// 之前
#[allow(unsafe_op_in_unsafe_fn)]
unsafe fn send_id(object: *mut c_void, selector_name: &str) -> *mut c_void {
    msg_send![object, sel!(selector_name)]
}

// 之后
unsafe fn send_id(object: *mut c_void, selector_name: &str) -> *mut c_void {
    // SAFETY: selector_name is a valid Objective-C selector string, and object
    // is a valid NSPasteboard reference. msg_send! performs the FFI call.
    unsafe { msg_send![object, sel!(selector_name)] }
}
```

需要按 Rust 惯用方式添加 `// SAFETY:` 注释，说明每个 unsafe 操作的安全不变量。

> **注意**: `unsafe_op_in_unsafe_fn` 已在 Rust 2024 edition 中变为 deny-by-default 错误。如果项目未来迁移到 2024 edition，修复此问题是前提条件。

## 风险与注意事项

1. **排序逻辑修改** (Step 3b) 是最关键的改动 — 修复 boost 遗漏的 bug 后排序行为会变化，需确认新行为符合预期
2. **Re-export 清理** (Step 2) 可能涉及大量 import 路径迁移，建议先确认消费者数量再评估工作量
3. **launcher.rs 拆分** (Step 5) 需确保模块之间的循环依赖得到正确处理
4. **unsafe 清理** (Step 8) 不影响功能，但 `// SAFETY:` 注释需准确描述安全不变量
5. **window_controller.rs 拆分** (Step 6) 相对简单，756 行拆为 4 个文件，风险低
6. 所有改动完成后必须运行 `cargo test --workspace` 和 `cargo clippy --workspace -- -D warnings`
